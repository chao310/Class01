# 用户信息管理平台 — 安全漏洞修复说明

> 本文档详细记录了原始代码中存在的安全漏洞以及对应的修复方案。
> 修复后的代码遵循 Flask 安全最佳实践，适用于开发与生产环境。

---

## 漏洞清单及修复

### 漏洞 1：明文密码存储（严重）

**问题描述：**
`USERS` 字典中的密码以明文形式直接存储，数据库（或代码）一旦泄露，所有用户的密码将完全暴露。

**修复方案：**
使用 `werkzeug.security.generate_password_hash()` 在应用启动时对密码进行哈希处理，并用 `check_password_hash()` 进行验证。

**代码变更（app.py）：**
```python
# 修复前
"admin": {"password": "admin123", ...}

# 修复后
from werkzeug.security import generate_password_hash, check_password_hash
"admin": {"password": generate_password_hash("admin123"), ...}

# 验证时
check_password_hash(USERS[username]["password"], password)
```

---

### 漏洞 2：密码泄露到前端页面（严重）

**问题描述：**
`index.html` 第 10 行 `{{ user.password }}` 将用户的密码明文渲染在页面上，任何可以看到该页面的人（或有 XSS 漏洞时任何脚本）都能获取密码。

**修复方案：**
1. 新增 `_safe_user_info()` 辅助函数，在将用户信息传递给模板前移除 `password` 字段。
2. 移除 `index.html` 中的密码展示行。

**代码变更（app.py）：**
```python
def _safe_user_info(username: str) -> dict | None:
    raw = USERS.get(username)
    if raw is None:
        return None
    return {k: v for k, v in raw.items() if k != "password"}
```

**代码变更（index.html）：**
```html
<!-- 修复前 -->
<li><span class="info-label">密码：</span>{{ user.password }}</li>

<!-- 修复后 -->
<!-- 该行已移除，密码不再展示 -->
```

---

### 漏洞 3：弱 Secret Key（高危）

**问题描述：**
`app.secret_key = "dev-key-2025"` 是一个硬编码的弱密钥，攻击者可轻易猜测并使用它伪造 Session Cookie，从而冒充任意用户。

**修复方案：**
使用环境变量或 `secrets.token_hex(32)` 生成不可预测的 64 字符十六进制密钥。

**代码变更（app.py）：**
```python
# 修复前
app.secret_key = "dev-key-2025"

# 修复后
app.secret_key = os.environ.get("SECRET_KEY") or secrets.token_hex(32)
```

---

### 漏洞 4：HTML 注释泄露默认凭据（高危）

**问题描述：**
`login.html` 顶部 HTML 注释中写明了默认管理员用户名和密码，任何查看页面源码的人都能直接获取管理员权限。

**修复方案：**
删除包含具体凭据的注释，替换为通用的信息提示。

**代码变更（login.html）：**
```html
<!-- 修复前 -->
<!-- 调试信息 - 默认管理员账号 用户名: admin 密码: admin123 -->

<!-- 修复后 -->
<!--
    调试信息 - 该页面不包含任何默认凭据
    所有用户信息仅在服务端通过安全哈希验证
-->
```

### 漏洞 5：缺少 CSRF 保护（中危）

**问题描述：**
登录表单没有 CSRF Token，攻击者可构造恶意页面诱导已登录用户（或利用跨站请求伪造）提交登录请求。

**修复方案：**
1. 实现基于 Session 的 CSRF Token 机制。
2. 在登录表单中添加隐藏的 `csrf_token` 字段。
3. 添加 `csrf_required` 装饰器校验所有 POST 请求。

**代码变更（app.py）：**
```python
def _generate_csrf_token() -> str:
    if "csrf_token" not in session:
        session["csrf_token"] = secrets.token_hex(16)
    return session["csrf_token"]

def csrf_required(f):
    @wraps(f)
    def decorated(*args, **kwargs):
        if request.method == "POST":
            token = request.form.get("csrf_token")
            if not token or not secrets.compare_digest(session.get("csrf_token"), token):
                abort(403)
        return f(*args, **kwargs)
    return decorated
```

**代码变更（login.html）：**
```html
<form action="/login" method="post" class="login-form">
    <input type="hidden" name="csrf_token" value="{{ csrf_token }}">
    <!-- 其他表单字段 -->
</form>
```

### 漏洞 6：缺少登录频率限制（中危）

**问题描述：**
登录接口没有任何速率限制，攻击者可对用户账户进行暴力密码破解。

**修复方案：**
基于 IP 地址实现内存内的登录频率限制：同一 IP 在 60 秒内最多尝试 5 次。

**代码变更（app.py）：**
```python
LOGIN_RATE_LIMIT = 5
LOGIN_RATE_WINDOW = 60
_login_attempts = {}

def _check_login_rate_limit(ip: str) -> bool:
    now = time.time()
    window_start = now - LOGIN_RATE_WINDOW
    if ip in _login_attempts:
        _login_attempts[ip] = [t for t in _login_attempts[ip] if t > window_start]
        if len(_login_attempts[ip]) >= LOGIN_RATE_LIMIT:
            return True
    return False
```

> **注意：** 内存方案在单进程 / 单机场景有效。多进程 / 分布式环境应使用 Redis 等集中式存储。

### 漏洞 7：缺少 Session Cookie 安全配置（中危）

**问题描述：**
未设置 Session Cookie 的 `HttpOnly` 和 `SameSite` 属性，存在被 JavaScript 读取（XSS 攻击）或被 CSRF 利用的风险。

**修复方案：**
通过 Flask 配置项设置 Cookie 安全属性。

**代码变更（app.py）：**
```python
app.config.update(
    SESSION_COOKIE_HTTPONLY=True,     # 禁止 JavaScript 读取 Cookie
    SESSION_COOKIE_SAMESITE="Lax",    # 限制跨站请求携带 Cookie
    SESSION_COOKIE_SECURE=False,      # 生产环境 HTTPS 时应设为 True
)
```

### 漏洞 8：Debug 模式默认开启（中危）

**问题描述：**
`app.run(debug=True)` 在生产环境中会暴露详细的错误堆栈信息（包含源码路径、变量值等），并可能开启 Werkzeug 调试器控制台。

**修复方案：**
debug 模式改为由环境变量 `FLASK_DEBUG` 控制，默认关闭。

**代码变更（app.py）：**
```python
# 修复前
app.run(host="0.0.0.0", port=5000, debug=True)

# 修复后
debug_mode = os.environ.get("FLASK_DEBUG", "0") == "1"
app.run(host="0.0.0.0", port=5000, debug=debug_mode)
```

---

### 漏洞 9：缺少输入验证（低危）

**问题描述：**
从表单获取的用户名和密码没有经过任何清洗或校验，可能包含恶意 payload。

**修复方案：**
对输入进行 `strip()` 去除首尾空白，并校验非空。

**代码变更（app.py）：**
```python
username = (request.form.get("username") or "").strip()
password = request.form.get("password") or ""

if not username or not password:
    return render_template("login.html", error="请输入用户名和密码", ...)
```

---

### 漏洞 10：Session 未在登录时重新生成（低危）

**问题描述：**
登录成功后未调用 `session.clear()` 或 `session.regenerate()`，存在 Session Fixation 攻击风险——攻击者可先让用户使用一个已知 Session ID，然后诱导用户登录。

**修复方案：**
登录成功时先清除旧 Session，再设置新数据并重新生成 CSRF Token。

**代码变更（app.py）：**
```python
# 登录成功分支
session.clear()
session["username"] = username
session["csrf_token"] = secrets.token_hex(16)
```

## 修复前后对比总表

| # | 漏洞类别 | 严重程度 | 修复前 | 修复后 |
|---|---------|---------|--------|--------|
| 1 | 密码存储 | 🔴 严重 | 明文存储 | `generate_password_hash` 哈希存储 |
| 2 | 密码泄露 | 🔴 严重 | 页面渲染 `{{ user.password }}` | `_safe_user_info()` 剥离密码字段 |
| 3 | Secret Key | 🟠 高危 | 硬编码 `"dev-key-2025"` | 环境变量或 `secrets.token_hex(32)` |
| 4 | 凭据泄露 | 🟠 高危 | HTML 注释明文暴露账号密码 | 删除具体凭据，改为通用注释 |
| 5 | CSRF | 🟡 中危 | 无 CSRF Token | 基于 Session 的 Token 校验 |
| 6 | 暴力破解 | 🟡 中危 | 无频率限制 | IP 维度 5 次/60 秒 限流 |
| 7 | Cookie 安全 | 🟡 中危 | 无 HttpOnly/SameSite | `HttpOnly=True`, `SameSite=Lax` |
| 8 | Debug 模式 | 🟡 中危 | `debug=True` 硬编码 | 环境变量控制，默认关闭 |
| 9 | 输入验证 | 🟢 低危 | 无 | `strip()` + 非空校验 |
| 10 | Session 固定 | 🟢 低危 | 未重置 Session | 登录成功时 `session.clear()` |

python app.py

SECRET_KEY="your-strong-secret-here" python app.py
