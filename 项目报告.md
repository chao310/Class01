## 一、项目概述

基于 Python Flask 框架开发的简易用户信息管理平台，包含**用户登录、注册、搜索**三大核心功能。项目以"先构建、再加固、后扩展"的思路进行迭代开发：

1. **Phase 1** — 构建基础登录功能（含多种刻意留下的安全漏洞）
2. **Phase 2** — 安全审计与修复（修复 10 个安全漏洞）
3. **Phase 3** — 功能扩展（增加注册和搜索，含可注入的 SQL 漏洞）

## 三、功能详情

### 3.1 用户登录

| 项目 | 说明 |
|------|------|
| 路由 | `GET/POST /login` |
| 认证方式 | Session + 内存字典 `USERS` |
| 密码存储 | Werkzeug `generate_password_hash()` 哈希存储 |

| 默认账号 | `admin / admin123`、`alice / alice2025` |

| 项目 | 说明 |
|------|------|
| 路由 | `GET/POST /register` |
| 数据存储 | SQLite 数据库 `data/users.db` |
| 密码存储 | ⚠️ **明文存储（SQL 注入教学演示）** |
| SQL 构建 | ⚠️ **f-string 字符串拼接（无参数化查询）** |

### 3.3 用户搜索

| 项目 | 说明 |
|------|------|
| 路由 | `GET /search?keyword=xxx` |
| 搜索范围 | 用户名 + 邮箱 LIKE 模糊匹配 |
| 结果展示 | ID / 用户名 / 邮箱 / 手机 表格形式 |
| SQL 构建 | ⚠️ **f-string 字符串拼接（无参数化查询）** |
| 调试输出 | 后台打印完整 SQL 语句到控制台 |

## 四、安全审计报告

### 4.1 原始版本漏洞（Phase 1 → Phase 2 修复）

| # | 漏洞类别 | 严重程度 | 修复方案 |
|---|---------|---------|---------|
| 1 | 明文密码存储 | 🔴 严重 | `generate_password_hash()` 哈希替代明文 |
| 2 | 密码泄露到前端页面 | 🔴 严重 | `_safe_user_info()` 剥离密码字段后再渲染 |
| 3 | 硬编码弱 Secret Key | 🟠 高危 | 环境变量或 `secrets.token_hex(32)` 随机生成 |
| 4 | HTML 注释泄露默认凭据 | 🟠 高危 | 删除包含账密的注释 |
| 5 | 缺少 CSRF 保护 | 🟡 中危 | 基于 Session 的 CSRF Token 校验 |
| 6 | 无登录频率限制 | 🟡 中危 | IP 维度 5次/60秒 内存限流 |
| 7 | Session Cookie 无安全配置 | 🟡 中危 | `HttpOnly=True`, `SameSite=Lax` |
| 8 | Debug 模式默认开启 | 🟡 中危 | `FLASK_DEBUG` 环境变量控制 |
| 9 | 缺少输入验证 | 🟢 低危 | `strip()` + 非空校验 |
| 10 | 登录未重置 Session | 🟢 低危 | 登录成功时 `session.clear()` |

### 4.2 新功能中的有意漏洞（Phase 3 — SQL 注入教学）

#### 漏洞 A：注册功能 SQL 注入

```python
# app.py 第 223 行 — 故意使用 f-string 拼接
query = f"INSERT INTO users (username, password, email, phone) VALUES ('{username}', '{password}', '{email}', '{phone}')"
```

**攻击示例：**
```bash
curl http://127.0.0.1:5000/register \
  -d "username=hacker', 'pass', 'h@x.com', '999')--&password=x&email=x&phone=x"
```

#### 漏洞 B：搜索功能 SQL 注入

```python
# app.py 第 246 行 — 故意使用 f-string 拼接
query = f"SELECT id, username, email, phone FROM users WHERE username LIKE '%{keyword}%' OR email LIKE '%{keyword}%'"
```

**攻击示例：**
```bash
# UNION 注入 — 返回自定义数据
curl "http://127.0.0.1:5000/search?keyword=%27%20UNION%20SELECT%201,'inj','inj@x.com','138'--"

# OR 注入 — 返回全部用户
curl "http://127.0.0.1:5000/search?keyword=%27%20OR%20'1'='1"
```

## 五、技术要点

### 5.1 安全编码最佳实践（登录模块）

```
密码存储    →  generate_password_hash() + check_password_hash()
CSRF 保护   →  secrets.compare_digest() 安全比对 Token
登录限流    →  滑动时间窗口 + IP 维度追踪
Secret Key  →  环境变量注入，默认随机 64 位十六进制
Session     →  HttpOnly + SameSite=Lax + 登录时重新生成
```

### 5.2 不安全的编码示例（注册/搜索模块，教学用）

```
密码存储    →  明文写入 SQLite
SQL 构建    →  f"SELECT ... WHERE ... LIKE '%{keyword}%'"
输入过滤    →  无任何转义或校验
错误信息    →  直接返回数据库异常信息
```

## 八、后续建议

1. **修复 SQL 注入**：注册和搜索改用参数化查询（`?` 占位符）
2. **注册密码加密**：写入 SQLite 前使用 `generate_password_hash()` 哈希
3. **数据库迁移**：登录认证也迁移到 SQLite，统一数据源
4. **部署配置**：设置 `SECRET_KEY` 环境变量，关闭 `FLASK_DEBUG`
5. **HTTPS 支持**：生产环境启用 `SESSION_COOKIE_SECURE=True`
6. **WAF / 输入过滤**：增加基础输入过滤层防止 XSS 和 SQL 注入
