# 文件上传漏洞分析与修复方案

> **项目：** 用户信息管理平台  
> **日期：** 2026-07-21  
> **涉及文件：** `app.py`、`templates/upload.html`

---

## 一、漏洞概述

当前 `/upload` 路由存在 **5 类安全漏洞**，攻击者可利用这些漏洞上传恶意文件、覆盖服务器文件、甚至获取服务器控制权。

| # | 漏洞类别 | 严重程度 | 位置 |
|---|---------|---------|------|
| 1 | 任意文件上传（无类型校验） | 🔴 严重 | app.py:281-283 |
| 2 | 路径遍历（Path Traversal） | 🔴 严重 | app.py:281-282 |
| 3 | 文件名覆盖（不生成唯一名） | 🟠 高危 | app.py:281-283 |
| 4 | 缺少 CSRF 保护 | 🟡 中危 | app.py:269-288 |
| 5 | 无文件内容校验（无 Magic Byte 检测） | 🟠 高危 | app.py:275-286 |

---

## 二、漏洞原理解析

### 漏洞 1：任意文件上传（Unrestricted File Upload）

**漏洞代码（app.py:280-283）：**
```python
# 使用用户上传的原始文件名保存（不重命名、不检查类型）
filename = file.filename
save_path = os.path.join(app.static_folder, "uploads", filename)
file.save(save_path)
```

**危害：** 没有任何文件类型校验，攻击者可上传：
- `.html` / `.htm` — 在服务器域下执行任意 HTML/JavaScript（存储型 XSS）
- `.svg` — SVG 内部可嵌入 JavaScript 脚本
- `.php` / `.phtml` — 如果服务器支持 PHP 则可执行任意代码
- `.py` — Python 脚本文件
- `.exe` / `.sh` — 可执行文件/脚本
- `.json` — 可能被其他应用读取

**攻击示例：**
```bash
# 上传一个包含恶意 JavaScript 的 HTML 文件
curl -X POST http://127.0.0.1:5000/upload \
  -b /tmp/cookies.txt \
  -F "file=@evil.html"

# 然后访问 http://127.0.0.1:5000/static/uploads/evil.html
# 浏览器会在服务器域下执行该 HTML，可窃取 Cookie
```

### 漏洞 2：路径遍历（Path Traversal）

**漏洞代码：**
```python
filename = file.filename
save_path = os.path.join(app.static_folder, "uploads", filename)
file.save(save_path)
```

**危害：** 文件名未经过 `secure_filename` 处理，攻击者可用 `../` 跳出上传目录：

```bash
# 文件名包含路径遍历符号
curl -X POST http://127.0.0.1:5000/upload \
  -b /tmp/cookies.txt \
  -F "file=@shell.py;filename=../../templates/index.html"
```

| 原始文件名 | `save_path` 实际路径 | 结果 |
|-----------|-------------------|------|
| `../../etc/passwd` | `static/uploads/../../etc/passwd` → `/etc/passwd` | 覆盖系统文件 ❌ |
| `../app.py` | `static/uploads/../app.py` | 覆盖应用文件 ❌ |
| `../../templates/index.html` | `static/uploads/../../templates/index.html` | 覆盖模板文件 ❌ |

### 漏洞 3：文件名覆盖（不生成唯一名）

**漏洞代码：**
```python
filename = file.filename
save_path = os.path.join(app.static_folder, "uploads", filename)
file.save(save_path)
```

**危害：** 使用原始文件名保存，相同文件名的上传会互相覆盖：

```bash
# 用户A上传 avatar.png（正常头像）
# 用户B上传 avatar.png（恶意文件）
# 用户A的头像被替换，且浏览器缓存可能导致用户A看到恶意内容
```

### 漏洞 4：缺少 CSRF 保护

**漏洞代码：** `/upload` 路由没有使用 `@csrf_required` 装饰器。

```python
@app.route("/upload", methods=["GET", "POST"])
def upload():
```

**危害：** 攻击者可在第三方网站上构造表单，诱导已登录用户提交上传请求：

```html
<!-- 攻击者构造的恶意页面 -->
<form action="http://127.0.0.1:5000/upload" method="post" enctype="multipart/form-data">
    <input type="file" name="file">
    <input type="submit" value="点我领红包">
</form>
```

### 漏洞 5：无文件内容校验（无 Magic Byte 检测）

**漏洞代码：** 未对文件内容做任何检测。仅依靠文件扩展名（甚至扩展名也不检查）是不够的。攻击者可以将 PHP shell 伪装成 `image.png`：

```bash
# 创建恶意文件（PNG 头 + PHP 代码）
echo -e '\x89PNG\r\n\x1a\n<?php system($_GET["cmd"]); ?>' > shell.php

# 上传时改为 .png 扩展名
curl -X POST http://127.0.0.1:5000/upload \
  -b /tmp/cookies.txt \
  -F "file=@shell.php;filename=shell.png"
```

## 三、POC 攻击验证

### POC 1：上传 HTML 恶意文件（存储型 XSS）

```bash
# 1. 登录
CSRF=$(curl -s -c /tmp/cookies.txt http://127.0.0.1:5000/login | grep -oP 'value="\K[a-f0-9]+(?=")')

  -d "csrf_token=$CSRF&username=admin&password=admin123"

echo '<html><body><script>alert(document.cookie)</script></body></html>' > /tmp/xss.html

# 3. 上传

  -F "file=@/tmp/xss.html"

# 4. 访问 http://127.0.0.1:5000/static/uploads/xss.html → 弹窗显示 Cookie
```

## 四、修复方案

### 综合修复要点

| # | 防护措施 | 方法 | 对应漏洞 |
|---|---------|------|---------|
| 1 | **文件名安全处理** | `werkzeug.utils.secure_filename()` | 路径遍历 |
| 2 | **文件扩展名校验** | 白名单 `.jpg .jpeg .png .gif .webp` | 任意文件上传 |
| 3 | **文件内容校验** | Magic Byte 检测（文件头签名） | 伪装文件上传 |
| 4 | **UUID 重命名** | `uuid.uuid4().hex` 生成唯一文件名 | 覆盖攻击 |
| 5 | **CSRF 保护** | 添加 `@csrf_required` 装饰器 | CSRF 上传 |
| 6 | **文件大小显式校验** | 读取前验证 Content-Length | 超大文件 DoS |
| 7 | **HTML 输出编码** | 模板中文件名使用 `{{ filename \| e }}` | XSS |

### 修复代码

#### 修复后 app.py — 上传路由部分

```python
import uuid

ALLOWED_EXTENSIONS = {"jpg", "jpeg", "png", "gif", "webp"}

# 常见图片文件的 Magic Bytes（文件头签名）
IMAGE_MAGIC_BYTES = {
    b"\x89PNG\r\n\x1a\n": "png",
    b"\xff\xd8\xff": "jpg/jpeg",
    b"GIF87a": "gif",
    b"GIF89a": "gif",
    b"RIFF": "webp",   # webp 以 RIFF 开头，需要进一步检测
}


def _allowed_file(filename: str) -> bool:
    """校验文件扩展名是否在白名单内"""
    if "." not in filename:
        return False
    ext = filename.rsplit(".", 1)[1].lower()
    return ext in ALLOWED_EXTENSIONS


def _validate_image_content(data: bytes) -> bool:
    """校验文件内容是否为真实图片（Magic Byte 检测）"""
    # 取文件前 8 字节用于匹配
    header = data[:8]
    for magic, fmt in IMAGE_MAGIC_BYTES.items():
        if header.startswith(magic):
            # WebP 特殊处理：检查第 8-11 字节是否为 "WEBP"
            if fmt == "webp":
                return data[8:12] == b"WEBP"
            return True

    # 额外检测 JPEG（需要检测第二个字节 0xE0/0xE1/0xFE...）
    if header[0:2] == b"\xff\xd8":
        return True

    return False


@app.route("/upload", methods=["GET", "POST"])
@csrf_required
def upload():
    """头像上传，需要登录"""
    if "username" not in session:
        return redirect("/login")

    if request.method == "POST":
        file = request.files.get("file")
        if file is None or file.filename == "":
            return render_template("upload.html", error="请选择要上传的文件")

        # [修复] 使用 secure_filename 防止路径遍历
        original_filename = file.filename
        safe_filename = secure_filename(original_filename)
        if not safe_filename:
            return render_template("upload.html", error="文件名不合法")

        # [修复] 校验文件扩展名（白名单模式）
        if not _allowed_file(safe_filename):
            return render_template(
                "upload.html",
                error="不支持的文件类型，仅允许上传 JPG/PNG/GIF/WebP 格式的图片",
            )

        # [修复] 读取文件内容进行 Magic Byte 校验
        file_data = file.read()
        if not _validate_image_content(file_data):
            return render_template("upload.html", error="文件内容不是有效的图片格式")

        # [修复] 生成唯一文件名（UUID），防止覆盖
        ext = safe_filename.rsplit(".", 1)[1].lower()
        unique_filename = f"{uuid.uuid4().hex}.{ext}"

        # [修复] 按用户分目录存储
        username = session["username"]
        user_upload_dir = os.path.join(app.static_folder, "uploads", username)
        os.makedirs(user_upload_dir, exist_ok=True)

        save_path = os.path.join(user_upload_dir, unique_filename)
        file.save(save_path)

        file_url = url_for("static", filename=f"uploads/{username}/{unique_filename}")
        return render_template("upload.html", file_url=file_url, filename=original_filename)

    return render_template("upload.html")
```

#### 修复后 upload.html — 前端增加 `accept` 限制

```html
<input type="file" id="file" name="file" class="file-input" accept="image/jpeg,image/png,image/gif,image/webp">
```

## 五、修复前后对比

### 5.1 文件名校验

| 对比项 | 修复前 | 修复后 |
|--------|--------|--------|
| 路径遍历 | ❌ `../../etc/passwd` → 写入系统目录 | ✅ `secure_filename()` → 移除危险字符 |
| 特殊字符 | ❌ `my<script>.png` → 原始保存 | ✅ → `my_script_.png` |
| Unicode | ❌ `头像.png` → 可能编码问题 | ✅ → `tou_xiang.png` |

### 5.2 文件类型校验

| 场景 | 修复前 | 修复后 |
|------|--------|--------|
| 上传 `shell.php` | ✅ 成功 | ❌ 拒绝（扩展名不在白名单） |
| 上传 `evil.html` | ✅ 成功 | ❌ 拒绝 |
| 上传 `shell.png`（含 PHP 代码） | ✅ 成功 | ❌ 拒绝（Magic Byte 不符） |
| 上传 `avatar.jpg` | ✅ 成功 | ✅ 成功 |
| 上传 `photo.png` | ✅ 成功 | ✅ 成功 |

### 5.3 防覆盖

| 场景 | 修复前 | 修复后 |
|------|--------|--------|
| 两次上传 `avatar.png` | 后者覆盖前者 | 生成不同 UUID，互不干扰 |
| 用户A 和 用户B 上传同名文件 | 互相覆盖 | 按用户名分目录存储 |

### 5.4 CSRF 保护

| 场景 | 修复前 | 修复后 |
|------|--------|--------|
| POST 请求缺少 CSRF Token | ✅ 接受 | ❌ 拒绝 (403) |
| POST 请求携带正确 CSRF Token | ✅ 成功 | ✅ 成功 |

## 六、防御文件上传最佳实践

| # | 最佳实践 | 说明 |
|---|---------|------|
| 1 | **白名单扩展名** | 只允许 `.{jpg,jpeg,png,gif,webp}`，拒绝所有其他扩展名 |
| 2 | **Magic Byte 检测** | 验证文件头签名，确保文件内容与扩展名一致 |
| 3 | **文件名净化** | 使用 `secure_filename()` 移除路径遍历和特殊字符 |
| 4 | **UUID 重命名** | 上传后生成唯一文件名，避免覆盖和信息泄露 |
| 5 | **CSRF 保护** | 所有写操作（POST/PUT/DELETE）都需校验 CSRF Token |
| 6 | **文件大小限制** | 服务器端和反向代理双重限制 |
| 7 | **隔离存储目录** | 按用户分目录，防止用户间文件互相影响 |
| 8 | **静态目录禁止执行脚本** | Nginx/Apache 配置 `static/` 目录不解析 PHP/Python |
| 9 | **Content-Type 校验** | `request.content_type` 辅助判断（不可单独依赖） |
| 10 | **定期清理** | 未使用的上传文件定期清理，减少攻击面 |

## 八、总结

| 漏洞 | 严重程度 | 修复方法 |
|------|---------|---------|
| 任意文件上传（无类型校验） | 🔴 严重 | 扩展名白名单 + Magic Byte 检测 |
| 路径遍历 | 🔴 严重 | `werkzeug.utils.secure_filename()` |
| 文件名覆盖 | 🟠 高危 | UUID 重命名 + 按用户分目录 |
| 缺少 CSRF 保护 | 🟡 中危 | 添加 `@csrf_required` 装饰器 |
| 无文件内容校验 | 🟠 高危 | Magic Byte 文件头签名检测 |

**核心原则：**
```
❌ 永远不要信任用户上传的文件名
❌ 永远不要相信文件扩展名
✅ 始终检查文件内容（Magic Byte）
✅ 始终使用白名单 + UUID 重命名
✅ 所有 POST 必须有 CSRF 保护
```
